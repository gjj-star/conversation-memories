## 自动知识捕获规则

当对话中出现以下情况时，自动执行：

### 触发条件

对话中产出了**有结构、可复用的知识**，且下次类似场景会直接用到的。覆盖以下类型：

#### A. 概念/方法论
- "XXX 是什么" → 定义、特点、适用场景
- "XXX 有什么优势和劣势" → 对比表
- "XXX 和 YYY 的关系" → 关系分析
- "XXX 的原理是什么" → 概念解释

#### B. 工具/项目评测
- "这个 XXX 怎么样" → 亮点、问题、评分、适用场景
- "XXX 的能力如何" → 能力边界评估（能做到什么 + 做不到什么）
- "XXX 对 YYY 的适配性" → 兼容性矩阵、改造点、限制

#### C. 方案选型与推荐
- "有没有好用的 XXX" → 候选列表 + 对比 + 推荐组合
- "用哪个方案好" → 方案对比 + 决策建议
- "A 和 B 选哪个" → 场景化推荐

#### D. 配置/安装/适配
- "给 XXX 安装/配置 YYY" → 安装路径、配置步骤、适配改动
- "XXX 在 YYY 上能用吗" → 兼容性结论 + 注意事项

#### E. 技术前沿与趋势
- "XXX 领域发展到什么程度" → 前沿总结、里程碑、关键玩家
- 搜索结果提炼出的领域全景

#### F. 知识蒸馏（来自外部内容）
- 用户给了链接/文章让我提取知识点 → 结构化保存

#### 判断标准（比类型标签更重要）

**存**：回答包含了对比表、步骤清单、决策树、评分/评级、配置参数、路径映射等结构化内容；下次类似场景需要用到同一份知识。
**不存**：单次操作指令、闲聊、情绪表达、已有记忆的简单复述、仅在当前项目上下文有效的临时决策。

不属于此范围：纯项目代码操作（改 bug、写功能）、单次工具调用、闲聊、已有回答的简单确认。

### 执行动作

1. 在回答完用户问题后，将核心知识提炼为 memory 文件，保存到 `~/.claude/projects/D--CC/memory/<topic-slug>.md`
2. 格式：frontmatter（name/description/type: reference）+ Markdown 正文
3. 更新 `MEMORY.md` 索引
4. Stop hook 会自动 git commit + push 到 https://github.com/gjj-star/conversation-memories.git

### 不重复写入

如果已有同名 memory 文件且内容覆盖了当前知识，跳过。如果新内容是对现有知识的补充（新发现、新观点），编辑已有文件追加。

**Why:** 对话中产出的知识如果不记录，下次对话就会丢失，需要重新推导。自动捕获让知识库持续成长。

**How to apply:** 每次回答完参考型问题后，检查 memory/ 目录是否已有覆盖，无则创建，有且不足则补充。
