# 知识图谱（Knowledge Graph）

## 定义

知识图谱 = 用「实体-关系-实体」三元组形式组织知识的图数据库，让机器能像人一样理解事物之间的关联，而不只是匹配关键词。

## 三元组结构

实体是节点，关系是边，一切知识表达为「主语-谓语-宾语」：

| 三元组 | 主语 | 谓语 | 宾语 |
|--------|------|------|------|
| 达芬奇创作了蒙娜丽莎 | 达芬奇 | 创作了 | 蒙娜丽莎 |
| 蒙娜丽莎收藏于卢浮宫 | 蒙娜丽莎 | 收藏于 | 卢浮宫 |
| 卢浮宫位于巴黎 | 卢浮宫 | 位于 | 巴黎 |

所有三元组装在一起形成图结构。

## 与 Ontology 的关系

参见 [[ontology-concept|Ontology 本体]]：
- **Ontology = Schema 层 / 概念骨架**：定义实体类型、关系类型、约束规则
- **Knowledge Graph = Instance 层 / 填入的实例数据**：按 Ontology 规则填入的具体三元组

类比：Ontology 是「Excel 表头设计」，知识图谱是「填了数据的行」。

## 两大类型

| 类型 | 开放知识图谱 | 企业知识图谱 |
|------|------------|------------|
| **代表** | Wikidata、DBpedia、Google Knowledge Graph | Amazon 商品图谱、LinkedIn 人才图谱、金融风控图谱 |
| **数据来源** | 从维基百科等公开源自动抽取 | 企业内部的数据库、文档、日志 |
| **用途** | 搜索引擎增强、智能问答 | 反欺诈（关联关系分析）、推荐系统、供应链可视化 |

## 技术栈

- **存储**：Neo4j / JanusGraph / Amazon Neptune（图数据库，非关系型）
- **查询语言**：SPARQL（W3C 标准）/ Cypher（Neo4j）/ Gremlin
- **构建方式**：结构化数据→直接映射（RDB→RDF）；半结构化→包装器解析；非结构化文本→NLP 抽取（NER 命名实体识别 + RE 关系抽取）

## 为什么用图数据库而非关系型数据库

关系型数据库查"达芬奇学生的学生的学生有谁"需自 JOIN 多次，复杂度随跳数爆炸；知识图谱就是从"达芬奇"节点沿"师从"边反向遍历三跳，O(3)。核心优势在**关联关系分析**：
- **反欺诈**：A 是 B 的法人 → B 持股 C → C 和 D 共享同一电话 → D 刚被标记欺诈，图里一秒看穿
- **供应链追溯**：一个零件出问题，沿图追溯所有包含该零件的产品批次
- **推荐系统**：「跟你看过同样电影的人也看了什么」→ 两条关联边

## 对比

| 传统数据库 | 知识图谱 |
|-----------|---------|
| 存"行和列" | 存"实体和关系" |
| 回答"什么值" | 回答"跟什么相关" |
| SQL 查表 | 图遍历查关系 |

**Why:** 与 [[ontology-concept|Ontology]] 构成知识组织的双层结构，是理解语义网、NLP 信息抽取、Palantir 类平台数据建模的基础。

**How to apply:** 遇到关联关系分析类问题（反欺诈、推荐、追溯）时优先考虑图模型；理解企业数据建模时区分 schema 层（Ontology）和 instance 层（KG）。
