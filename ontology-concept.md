Ontology（本体论/本体）有两层同源含义：

**1. 哲学本体论（源头）**：形而上学分支，研究"存在"本身——什么存在、存在意味着什么、万物如何分类（实体/属性/关系/事件）。词源：希腊语 ontos（存在者）+ logos（学说）。代表：亚里士多德《范畴篇》、海德格尔《存在与时间》、蒯因（"存在即成为约束变量的值"）。

**2. 计算机/AI 本体（借用义）**：Gruber 1993 经典定义——"对概念化的显式规范"（an explicit specification of a conceptualization）。即把某领域的概念体系用机器可处理的方式显式定义出来，四大构件：类（Class）、实例（Instance）、关系（Relation）、约束（Axiom）。例：电商本体定义"用户/商品/订单"类、"购买"关系、"订单必须有买家"约束。

**与知识图谱的关系**：本体 = 图谱的 schema/骨架层（概念与规则），知识图谱 = 填入的实例数据。技术栈：RDF/RDFS、OWL、SPARQL、Protégé。应用：语义网、数据集成、生物医学（Gene Ontology）、企业数据建模。

## Palantir Ontology 深度拆解

Palantir Foundry 的核心差异化能力——不是数据库、不是知识图谱、不是 ER 图，而是**把企业散落在各处的原始数据映射成业务人员可以直接理解和操作的数字孪生层**。

### 三要素（+ 一动作）

| 构件 | 含义 | 航空业举例 |
|------|------|-----------|
| **Object Type**（对象类型） | 业务概念 | 飞机（Aircraft）、发动机、维修工单 |
| **Property**（属性） | 描述字段 | 序列号、飞行里程、上次维修时间、实时振动值 |
| **Link**（关联） | 对象间关系 | 飞机 → 安装了 → 发动机；维修工单 → 针对 → 发动机 |
| **Action**（动作） | 可在对象上执行的操作 | 创建紧急维修工单、批准放行、标记停飞 |

属性值来自多个底层系统的自动映射——ERP 的资产表、维修系统的作业记录、IoT 传感器、航班调度 API——对用户表现为一个统一对象。

### 解决的根问题：企业数据碎片化

同一架飞机，ERP 叫 `ASSET_CN_4432`，维修系统叫 `AIRCRAFT_B737_NG_018`，调度系统叫 `B-5678`。传统方案是 ETL 建数据仓库（结果是另一个同样难维护的大表）；Palantir 的解法是**不移动数据，在数据上方架语义映射层**，底层数据留在原系统，Ontology 在逻辑层统一对象。

### 三大关键能力

1. **数据不动，语义统一**：不需要把数据倒进数据湖，逻辑层统一，底层该在哪还在哪
2. **一次建模，所有应用共享**：看板、指挥大屏、工单系统用的都是同一个 Ontology 对象，天然数据一致性
3. **Action 不只是"看"**：传统 BI 只能展示数据，Ontology Action 让用户直接在对象上触发跨系统操作——点击"创建紧急维修工单"同时写入维修系统和 ERP

### 与 FDE 的关系

**FDE 岗位本身就是 Palantir 发明的**。Palantir 部署工程师的核心流程：深入客户业务 → 理解数据散落位置 → 设计 Ontology（定义对象/属性/关联）→ 写 Pipeline 接数据 → 开发操作界面 → 培训客户。这解释了为什么 FDE 不是纯技术岗——设计 Ontology 需同时理解业务语义和技术实现，你在 AIBP 业财实习的数据导入和业务规则理解就是低配版 Ontology 搭建训练。详见 [[guojinjia-fde-plan]]、[[pm-vs-fde-comparison]]。

### AI 时代的 Ontology 回热

Palantir AIP（2023）核心思路是 **Ontology + LLM**：用户自然语言提问"哪些飞机下周需要大修？"→ LLM 理解意图 → Ontology 提供业务上下文（知道"飞机""大修""下周"对应什么对象/属性/计算逻辑）→ LLM 生成查询/调用 Action → 返回结果+可执行按钮。这解决了 LLM 在企业场景的最大痛点——幻觉和不知道上下文——也是 RAG、Text-to-SQL 语义层、Agentic AI 都在往本体驱动方向走的原因。

**一句话记忆**：本体 = 明确回答"这个领域/世界里有哪些东西、它们如何分类和关联"的形式化清单；哲学问的是整个世界，工程问的是某个业务域。Palantir 把这个哲学概念变成了十亿美元产品。
