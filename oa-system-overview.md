# OA 系统概述

## 定义

OA（Office Automation，办公自动化）是将企业内部事务性工作流程——审批、公文、考勤、会议、公告、文档——从纸质/口头/邮件搬上信息系统，实现**流程标准化、操作可追踪、决策有留痕**。

与已有概念的关联：
- [[danju-documents|单据]]：OA 是单据电子化的核心载体——报销单、借款单、出差申请单都在 OA 里填写和流转
- [[taizhang-ledger|台账]]：OA 流程数据自然形成某类台账（如公章使用台账、合同审批台账），省去手工维护
- [[agent-skill-automation|Agentic 自动化]]：OA 是 RPA/Agent 最密集的改造对象——填单、审批、跨系统搬运数据

## 核心模块

- **流程引擎**（心脏）：请假审批流 ≈ 申请人提交 → 直属上级审批 → 部门负责人审批 → HR 备案 → 抄送考勤系统。每一步该谁处理、超时催办、驳回退回，都由流程引擎定义。
- **公文管理**：收发文、公文流转、归档
- **考勤排班**：打卡、请假、加班记录
- **会议日程**：会议室预定、日程共享
- **文档库**：共享文档、版本管理
- **公告通知**：企业内部信息发布
- **即时通讯（IM）**：消息推送、内部沟通
- **权限管理（RBAC）**：角色/部门/岗位权限体系

## 技术架构（典型）

浏览器/移动端 → Nginx 反向代理 → 应用层（Java/Spring Boot 或 .NET，含流程引擎 Activiti/Flowable/自研、表单引擎、权限框架、消息中间件）→ 数据层（MySQL/PostgreSQL/Oracle）→ 文件存储（NAS/OSS/MinIO）。

几乎都是 B/S 架构，移动端（钉钉/飞书/企业微信小程序）是标配。

## 主流产品与选型

| 产品 | 定位 | 优势 | 劣势 | 适用规模 |
|------|------|------|------|---------|
| **泛微 e-cology** | 老牌专业 OA | 流程引擎极强，二次开发灵活 | 技术老旧（Java），界面老气 | 中大型企业 |
| **致远互联 A8** | 老牌专业 OA | 公文管理强，政务领域深耕 | 实施重，灵活性不如泛微 | 中大型/政府 |
| **蓝凌** | 知识管理型 OA | 文档/知识库管理突出 | 流程引擎一般 | 知识密集型企业 |
| **钉钉** | 平台型 | 免费/低成本，生态丰富 | 深度流程定制弱，数据在阿里云 | 中小企业 |
| **飞书** | 平台型 | 协作体验最佳，文档+审批一体 | 审批流复杂度有限 | 互联网/新经济 |
| **企业微信** | 平台型 | 连接微信生态（客户群/小程序） | 审批功能相对简陋 | 销售/服务型企业 |
| **明道云/简道云** | 零/低代码 | 业务人员可自建应用 | 复杂流程和大并发场景乏力 | 中小企业/部门级 |
| **自研 OA** | 定制 | 精确匹配需求 | 成本高、迭代慢 | 超大型集团 |

### 决策框架

决策树：先看规模（<100 人→钉钉/飞书标准版够用）→ 看流程复杂度（5 级以上串并行混合审批→必须专业 OA）→ 看集成需求（ERP 深度对接→泛微接口最完善）→ 看预算（泛微百人部署约 15-30 万：license+实施+首年运维，实施费通常是 license 的 1-3 倍）→ 看 IT 能力（无专职 IT→零代码平台自建）。

### 最被低估的选型因素
- **流程引擎版本管理**：业务规则会变，OA 流程能否像代码一样做版本控制和回归？很多产品不支持，一改永久覆盖
- **移动端体验**：制造业工人没有电脑，审批必须能在微信/钉钉里直接处理不跳转
- **离职交接成本**：OA 配置靠"老师傅"手搓，人走了流程设计思路也丢了——有没有流程文档自动生成能力？

**Why:** 用户（郭锦佳）正在通过 AIBP 业财实习进入 B2B 部署场景，OA 系统是其未来 FDE 工作中最高频接触的系统类型之一，需要从概念、架构、选型到实操建立完整认知。

**How to apply:** 遇到 OA 相关话题时以此为知识锚点；选型讨论时参考决策框架五步法；与 [[oa-fde-relationship]]、[[oa-agent-integration]] 配合使用。
